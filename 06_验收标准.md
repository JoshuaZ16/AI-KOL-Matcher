# 验收标准

| 验收项 | 合格标准 |
|--------|----------|
| 功能完整性 | 能完成从需求输入到推荐输出的完整流程，输出 TOP 10 达人列表 |
| 输出格式 | 输出包含排名、达人名称、平台、粉丝数、报价、匹配分数、推荐理由、预估 ROI、风险提示的完整表格 |
| 业务逻辑 | 推荐结果符合用户需求（受众匹配、预算范围内、平台正确），排序合理（高匹配度、高性价比排前面） |
| 理由说明 | 每个达人都有具体的推荐理由，引用数据支撑（粉丝数、互动率、转化率、受众匹配度） |
| 人工确认 | 标注风险提示（广告比例高/数据异常），提示用户需要人工复核最终投放决策 |
| Demo 表现 | 3-5 分钟内能展示完整流程：输入需求 → 查看推荐列表 → 查看推荐理由和投放建议 |
| 数据合理性 | 匹配分数计算逻辑合理，综合考虑受众/语义匹配、性价比、风险评估、粉丝规模、互动率、转化率等指标 |
| 多平台支持 | 能够筛选和推荐不同平台的达人（小红书、抖音、B站、微博），平台分布合理 |
| ROI 预估 | 能够根据历史转化率和报价计算预估 ROI，数值合理（如 1:2.5 表示投入 1 元回报 2.5 元） |
| 投放建议 | 输出总体投放建议，包括预算分配、平台组合、注意事项，建议具体可行 |
| AI 使用边界 | 排名由规则评分决定，LLM 只生成推荐理由、投放建议、风险提示等表达内容，不直接控制排序 |

---

## 详细验收说明

### 1. 功能完整性
- 用户输入需求后，系统能够自动筛选达人库
- 能够计算匹配分数并排序
- 能够输出 TOP 10 达人推荐列表
- 能够生成推荐理由和投放建议

### 2. 输出格式
- 表格包含所有必需字段（排名、达人名称、平台、粉丝数、报价、匹配分、推荐理由、ROI、风险提示）
- 表格格式清晰，易于阅读
- 投放建议包含预算分配、平台组合、注意事项三部分

### 3. 业务逻辑
- 推荐的达人符合用户指定的平台（如用户选择小红书，不应推荐抖音达人）
- 推荐的达人报价在用户预算范围内
- 推荐的达人内容领域与用户需求匹配
- 排序合理：高匹配度、高性价比、低风险的达人排在前面

### 4. 理由说明
- 每个达人的推荐理由必须具体，不能泛泛而谈
- 推荐理由必须引用数据（如"受众高度匹配，大学生占 85%"、"互动率 3.8%"）
- 推荐理由长度适中（50-100 字）

### 5. 人工确认
- 高风险达人（广告比例高、数据异常）必须在"风险提示"列标注
- 投放建议中必须提示"最终投放决策需人工复核"
- 对于首次合作的达人，建议人工核实数据真实性

### 6. Demo 表现
- 演示流程清晰：输入需求 → 系统处理 → 输出推荐列表 → 查看详细理由
- 演示时间控制在 3-5 分钟
- 能够展示 AI 的核心价值：自动筛选、多维度评分、数据驱动决策

### 7. 数据合理性
- 匹配分数在 0-100 分之间
- 分数计算逻辑透明：脚本兼容基础权重，服务化推荐按拉新、转化、曝光、种草等推广目标动态调整权重
- 标签与语义匹配可解释：推荐详情能看到达人标签和语义命中标签
- 排序结果符合直觉（高分排前面）
- 同一输入数据和同一评分规则下，TOP 10 排名应稳定可复现

### 8. 多平台支持
- 能够处理小红书、抖音、B站、微博四个平台的达人数据
- 推荐列表中平台分布合理（不应全部集中在一个平台）
- 投放建议中能够给出平台组合建议

### 9. ROI 预估
- ROI 数值合理（通常在 1:1.5 到 1:5 之间）
- ROI 计算基于历史转化率和报价
- ROI 数值能够帮助用户判断投放性价比

### 10. 投放建议
- 预算分配建议具体（如"优先选择排名前 3 的达人，占预算 60%"）
- 平台组合建议合理（如"小红书适合图文种草，抖音适合短视频展示"）
- 注意事项实用（如"排名第 1 的达人有广告比例高风险，建议要求原创内容"）

### 11. AI 使用边界
- LLM 不直接生成匹配分数或决定排名
- LLM 文案必须围绕已排序的 TOP 10 解释原因和建议
- 如接入自然语言需求理解，LLM 只负责把模糊需求转成结构化字段，再进入规则评分流程
